Technical Protocol

How It Works Under the Hood

Domain Connect is an HTTP-based application protocol. You implement two things: a discovery record in DNS and a web endpoint that handles template application requests.

👤
End User
Domain owner, no DNS knowledge required
  • Initiates from service UI
  • Authenticates at your portal
  • Reviews & approves changes
domain name
⚙️
Service Provider
SaaS services, email providers, website builders…
  • Looks up _domainconnect TXT
  • Fetches discovery document
  • Redirects user + template vars
redirect + template
🏗️
Your Platform
DNS Provider / Registrar
  • Hosts _domainconnect TXT record
  • Exposes discovery endpoint
  • Authenticates user (your SSO)
  • Shows consent UI (your design)
  • Applies template to zone
1

DNS Discovery

You publish a TXT record at _domainconnect.yourdomain.com pointing to your API endpoint. Service providers look this up to find your Domain Connect endpoint.

2

Discovery Document

Your endpoint returns a JSON document advertising which templates you support, your OAuth endpoints, and other capabilities.

3

Synchronous Flow

For one-off setup: user is redirected to your consent URL with template ID + variable values in query params. You authenticate, show consent, apply changes.

4

Asynchronous Flow (OAuth)

For ongoing management: service provider gets an OAuth token scoped to a specific template. It can call your API to update records as IPs change or new subdomains are added.

5

Template Enforcement

You only apply changes described in the pre-approved template. SPF merging, conflict resolution, and ephemeral record cleanup are all handled by protocol rules you implement once.

6

URL Signature Verification

Service providers can sign redirect URLs so you can verify requests haven't been tampered with before applying any changes.

Templates

What You're Applying: A Template

A Domain Connect template is a JSON file published by a service provider that declares exactly which DNS records to create, update, or remove. You apply it — you don't write it.

example-service-template.json 
{
  "providerId": "yourservice.example",
  "serviceId": "website",
  // --- Metadata shown in your UI ---
  "providerName": "Cool Hoster",
  "serviceName": "Website",
  "logoUrl": "https://…/logo.png",

  // --- DNS records to apply ---
  "records": [
    {
      "type": "A",
      "host": "@",
      "pointsTo": "%IP%",
      "ttl": 3600
    },
    {
      "type": "CNAME",
      "host": "www",
      "pointsTo": "hosting.yourservice.example",
      "ttl": 3600
    },
    {
      // Ephemeral – safe to remove after template is applied
      "type": "TXT",
      "host": "@",
      "data": "%VERIFY_TOKEN%",
      "ttl": 300,
      "essential": "OnApply"
    }
  ]
}
  • Identification
    providerId + serviceId uniquely identify the template. You look this up when a request arrives to decide whether to proceed.
  • Metadata
    serviceName, description and logoUrl are what you show in your consent UI so users know what they're approving.
  • Records
    The exact DNS record types, hosts, and values to apply. %IP% and similar tokens are filled in from query-string variables passed in the redirect URL.
  • Conflict
    Optional fields like txtConflictMatchingMode tell you how to handle existing records — replace, skip, or merge (important for SPF records).
  • Ephemeral
    Records marked as ephemeral can be safely removed after the template has been applied, without triggering any conflict rules.

Implementation Guide

How to Get Started

Implementation typically takes 1–3 sprints depending on your existing DNS management API. The community and open-source reference code will accelerate you significantly.

1

Read the Specification

Start with the IETF draft (draft-kowalik-domainconnect). It defines the discovery record format, discovery document schema, sync and async flows, and template semantics.

→ IETF DCONN Working Group
2

Publish the Discovery TXT Record

Add a TXT record for _domainconnect at your managed domains pointing to your API hostname. This is how service providers find your endpoint.

→ Getting Started Guide
3

Build the Discovery Endpoint

Implement GET /v2/domainTemplates/providers/{providerId} — returns JSON advertising your capabilities, OAuth endpoints, and template support.

→ Reference Implementations on GitHub
4

Implement the Sync Flow

Handle redirect URLs containing template + variable parameters. Authenticate the user, render your consent UI showing the requested changes, then apply the template to the DNS zone.

5

Add OAuth for Async Flow (Optional)

For service providers that need ongoing management (e.g., updating IPs), implement the OAuth token flow scoped to individual templates.

6

List Your Platform

Add yourself to the domainconnect.org DNS Providers page and notify major service providers. You'll start appearing in their "Connect automatically" flows.

→ DNS Providers List
