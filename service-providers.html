Template Guide

Writing Your Domain Connect Template

A template is a JSON file you publish to the open template registry. It's the only thing that describes what DNS changes your service needs — DNS providers use it to apply the right records.

your-service-template.json 
{
  // Who you are
  "providerId": "yourservice.example",
  "serviceId": "email",
  "providerName": "Your Service Inc.",
  "serviceName": "Business Email",
  "version": 1,

  // Shown to users in consent UI
  "logoUrl": "https://yourservice.example/logo.png",
  "description": "Configure email for your domain",

  // Where users return after setup
  "syncRedirectDomain": "yourservice.example",

  // The DNS records you need
  "records": [
    {
      "type": "MX",
      "host": "@",
      "pointsTo": "mail.yourservice.example",
      "priority": 10,
      "ttl": 3600
    },
    {
      "type": "TXT",
      "host": "@",
      "data": "v=spf1 include:spf.%IP% ~all",
      // Merge with existing SPF records
      "txtConflictMatchingMode": "Prefix",
      "txtConflictMatchingPrefix": "v=spf1",
      "ttl": 3600
    },
    {
      // Ephemeral – safe to remove after template is applied
      "type": "TXT",
      "host": "@",
      "data": "%VERIFY%",
      "ttl": 300,
      "essential": "OnApply"
    }
  ]
}
🆔

Provider & Service Identity

providerId should be your domain name. serviceId is a short identifier for this specific integration (e.g., "email", "website", "ssl"). Together they uniquely identify your template.

🖼️

Display Metadata

Your logoUrl, providerName, and serviceName appear in the DNS provider's consent UI. Keep them clear and recognizable so users trust the authorization screen.

📝

Variable Substitution

Use %VARNAME% tokens for values you pass at runtime (IP addresses, verification tokens, user-specific values). These are passed as query parameters in your redirect URL.

🔀

SPF Merging & Conflict Resolution

Use txtConflictMatchingMode: "Prefix" for SPF records to ensure your record is merged with existing SPF — not overwriting a customer's other email service settings.

⏱️

Ephemeral Records

Records marked as ephemeral can be removed after the template has been applied, without triggering any conflict rules. This makes them safe to clean up at any point — the DNS provider deletes them automatically once the template application is complete, keeping zones tidy.

Implementation Guide

How to Add Domain Connect to Your Service

Implementation is straightforward — your main task is writing a template. The DNS providers handle the hard parts: authentication, consent UI, and DNS zone management.

1

Understand the Specification

Read the IETF draft (draft-kowalik-domainconnect) to understand the full protocol — especially the discovery flow, sync vs. async flows, and how template variables work.

→ IETF DCONN Working Group
2

Write Your Template

Create a JSON template file declaring the DNS records your service needs. Study existing templates in the public registry as a reference — they cover a wide range of service types.

→ Template Registry (GitHub)
3

Submit to Template Registry

Open a pull request to the domain-connect/Templates GitHub repository. Your template will be available to all DNS providers once merged.

→ Submit Your Template
4

Implement the Discovery Lookup

When a user enters their domain, query the DNS for _domainconnect.{domain} TXT record. If present, fetch the discovery document from that endpoint to get the DNS provider's API details.

5

Build the Redirect & Callback

Construct the redirect URL with your template ID and runtime variable values (IP addresses, verification tokens). Handle the callback URL after the user approves the changes.

→ Reference SDK Code
6

(Optional) Implement OAuth for Updates

For services that update infrastructure regularly, request an OAuth token during setup. Use it to push DNS changes directly to customer zones without requiring user re-authorization.
